# Phase 999.13: Sustentação — Intel Operacional por Pessoa - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Inteligência operacional da operação de sustentação: vazão de tickets (in vs out), curva histórica de backlog, top tipos por frequência, detecção de recorrência (candidatos a resolver na raiz). Visualização na SustentacaoView existente.

**Reframing importante:** O nome original "por pessoa" pressupunha múltiplos liderados. Na prática, o gestor trabalha com 1 liderado em sustentação — o foco real é **controle da operação**, não comparação entre assignees.

Escopo: vazão, tipos, recorrência visíveis na UI. NÃO inclui: notificações proativas (999.14), alertas por threshold (999.14).

</domain>

<decisions>
## Implementation Decisions

### Vazão de tickets
- **D-01:** Exibir in/out semanal (tickets abertos na semana vs resolvidos) — mostra se a operação está ganhando ou perdendo terreno.
- **D-02:** Exibir curva histórica de backlog (tickets abertos ao longo do tempo) — tendência de crescimento ou redução da fila. Usar data do history.json existente (granularidade diária, 90 dias).

### Top tipos + recorrência
- **D-03:** Ranking de tipos por frequência (todos os tickets abertos, não só breach). Já existe lógica de tipoCounts em SupportBoardClient — estender para incluir tickets fechados nos últimos 30d também.
- **D-04:** Detecção de recorrência: se o mesmo tipo+label aparece >2x em 30 dias, é candidato a resolver na raiz. Regra determinística, sem IA nesta fase.
- **D-05:** Recorrentes exibidos com destaque visual ("⚠️ Candidato a raiz — X ocorrências em 30d").

### Visualização
- **D-06:** Nova seção "Inteligência Operacional" na SustentacaoView, abaixo dos cards de SLA compliance. Sem nova aba — mesma tela.
- **D-07:** Seção contém: (1) gráfico in/out semanal, (2) curva de backlog histórico, (3) top tipos com freqüência, (4) lista de recorrentes detectados.

### Fronteira com 999.14
- **D-08:** Esta fase entrega dados e padrões visíveis na UI. 999.14 entrega notificações proativas (app avisa quando threshold cruza). Sem badges de alerta ou notificações aqui.

### Claude's Discretion
- Biblioteca de gráfico para in/out e backlog (pode reusar SVG inline do 999.11 ou recharts se já adotado)
- Número de semanas exibidas no gráfico in/out (sugestão: 8 semanas)
- Layout exato dos sub-elementos dentro da seção
- Se exibir "total de recorrentes" como número no header da seção

</decisions>

<code_context>
## Existing Code Insights

### Dados já disponíveis
- `issues` (JiraIssue[]) buscados dos últimos 90d com: `key`, `summary`, `type`, `labels`, `assignee`, `created`, `resolved`, `statusCategory`, `status`
- `tipoCounts` já calculado para tickets abertos — só precisa estender para fechados também
- `history.json` tem `ticketsAbertos` por dia → curva de backlog já tem dados, só falta visualização

### O que precisa ser calculado/adicionado
- **In/out semanal:** agrupar issues por semana de criação (in) e semana de resolução (out) — dos últimos 90d
- **Recorrência:** para cada combinação tipo+label, contar ocorrências nos últimos 30d; flaggar se >2
- **SupportBoardSnapshot** precisa de novos campos: `inOutSemanal[]`, `recorrentesDetectados[]`

### Padrões estabelecidos
- Cache com readFileSync/writeFileSync (sem async fs)
- IPC retorna dados processados ao renderer — cálculos ficam no main process
- SVG inline para mini charts (999.11 já estabeleceu o padrão)

</code_context>

<specifics>
## Specific Notes

- Usuário trabalha com **1 liderado** em sustentação — intel "por assignee" não se aplica como feature de comparação. O valor é na operação em si.
- Core value: "controle da operação e me antecipar" — dados históricos + padrões são o diferencial vs o que já existe (snapshot instantâneo).
- "Frequência de tipos a ponto de entender o que dói mais e avaliar automações" — recorrência detectada é diretamente acionável para decisões de automação.

</specifics>

## Deferred Ideas

- **Notificações proativas** — "fila crescendo há 3 semanas consecutivas" → pertence à Phase 999.14 (alertas proativos).
