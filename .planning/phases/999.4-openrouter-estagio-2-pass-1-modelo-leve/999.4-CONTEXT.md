# Phase 999.4: OpenRouter Estágio 2 — Pass 1 com modelo leve — Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrar o Pass 1 (identificação de `pessoa_principal` e metadados básicos do artefato) para OpenRouter quando o modelo híbrido estiver ativo. Pass 1 é o candidato ideal: sem contexto de perfil, output estruturado, e maior multiplicador de latência por rodar em todo artefato ingestionado. Padrão de roteamento condicional com fallback para Claude CLI já estabelecido pela 999.3 (Pass Cerimônia).

**Fora do escopo desta fase:**
- Pass 2 (com perfil) — qualidade crítica, permanece no Claude CLI
- Pass 1:1 deep — já roda em `haiku`, latência aceitável, qualidade é o que importa
- Novo UI ou mudança de toggle na SettingsView
- Métricas ou relatório de performance na UI

</domain>

<decisions>
## Implementation Decisions

### Controle de ativação
- **D-01:** Mesmo toggle `useHybridModel` existente — quem ativou o híbrido para Pass Cerimônia ativa automaticamente para Pass 1 também. Sem mudança na UI e sem novo campo no AppSettings.

### Adequação do prompt ao modelo leve
- **D-02:** Não modificar o prompt do Pass 1 (`buildIngestionPrompt`). O ajuste é feito no payload da chamada OpenRouter: adicionar campo `system` com instrução explícita de output JSON — ex: `"You must respond with valid JSON only, no markdown code blocks, no explanation."`. Isso é necessário porque o Claude CLI usa `--output-format json` implicitamente; a API REST do OpenRouter retorna texto livre.
- **D-03:** O `validateIngestionResult` existente atua como filtro de qualidade. Se o modelo leve retornar schema inválido (campos ausentes, tipos errados), o mecanismo de fallback para Claude CLI é acionado automaticamente — a validação já é a proteção.

### Critério de elegibilidade por tamanho
- **D-04:** Sem threshold por tamanho de artefato — OpenRouter é usado para todo Pass 1 quando híbrido ativo, independente do tamanho do artefato. O fallback automático via `validateIngestionResult` é proteção suficiente. Threshold manual seria duplicar a proteção com número arbitrário.

### Comportamento de fallback
- **D-05:** Padrão idêntico ao Pass Cerimônia (999.3): tentar OpenRouter → se falhar (HTTP error, network error ou schema inválido após validação) → fallback para Claude CLI com log de warning → se Claude CLI também falhar → throw Error (comportamento atual inalterado). Pass 1 é blocking — não pode ser silenciado.

### Claude's Discretion
- Texto exato da instrução `system` no payload OpenRouter (desde que enforce JSON puro sem markdown)
- Onde exatamente no `runOpenRouterPrompt` ou no caller aplicar o campo `system`
- Timeout para Pass 1 via OpenRouter (expectativa: menor que 90_000ms do Claude CLI, mas Claude decide o valor seguro)

</decisions>

<specifics>
## Specific Ideas

- A instrução de output JSON deve prevenir o padrão comum de modelos leves de envolver a resposta em ```json ... ``` — isso quebraria o parser atual
- O modelo padrão (`google/gemma-3-27b-it`) já foi validado no Pass Cerimônia — usar o mesmo para Pass 1 na primeira iteração

</specifics>
