# Phase 999.4: Gemini Preprocessing Pass

## Contexto

A ingestão de 1:1 está consumindo muito do Claude Code CLI — três 1:1s consumiram ~35% do consumo do período. O problema são 3 passes que recebem o transcript na íntegra:

1. **Pass 1** — identifica pessoa (90s, haiku)
2. **Pass 2** — com perfil.md completo (180s, haiku)
3. **1:1 Deep Pass** — transcript + perfil + config + actions + health history (300s) ← o vilão principal

A transcrição bruta de voz tem muito ruído: repetições, "né", "tipo", interjeições, frases incompletas. Um pré-processamento com **Gemini Flash** antes de qualquer pass reduziria o input em todos os três.

## Solução Proposta

**Pipeline novo:**
```
Transcript bruto (10–30K chars)
    ↓ Pass 0: Gemini Flash (Google AI API)
Transcript limpo/estruturado (2–5K chars)
    ↓ Pass 1 → Pass 2 → 1:1 Deep (Claude)
```

**Economia estimada:** ~60-70% de redução de tokens nos passes subsequentes.

## Escopo

### Must-Have (MVP)

1. Integração Google AI API para Gemini Flash
2. Campo `googleAiApiKey` em AppSettings
3. Função `runGeminiPreprocessing()` em novo arquivo
4. Prompt de pré-processamento para limpar transcrições
5. Integração no início do pipeline (antes do Pass 1)
6. Toggle em SettingsView para ativar/desativar pré-processamento
7. Métricas de redução de tokens (log)

### Nice-to-Have (Futuro)

- Configuração de modelo Gemini (Flash vs Pro)
- Fallback para OpenRouter se Google AI falhar
- Cache de transcrições pré-processadas

## Fluxo de Dados

```
File dropped → IngestionPipeline.enqueue()
    ↓
processItem():
    - Ler arquivo
    - SE useGeminiPreprocessing && googleAiApiKey:
        - Chamar preprocessTranscript(textoBruto)
        - Log: "Transcript reduzido de X para Y chars (Z% economia)"
        - Usar textoLimpo nos passes seguintes
    - SENÃO: usar textoBruto (comportamento atual)
    ↓
Pass 1 (identificação) → textoLimpo
Pass 2 (perfil) → textoLimpo  
1:1 Deep Pass → textoLimpo + contexto adicional
```

## Prompt de Pré-Processamento

**Objetivo:** Limpar ruído, remover repetições, estruturar em seções claras.

**Input:** Transcrição bruta (VOICE ou texto)
**Output:** Texto limpo estruturado:
- Seção "Participantes" (quem falou)
- Seção "Discussão" (pontos principais, sem repetições)
- Seção "Decisões" (o que foi acordado)
- Seção "Ações" (quem faz o quê, até quando)

**Instruções ao modelo:**
- Remover preenchidos ("né", "tipo", "entendeu")
- Agrupar ideias fragmentadas em frases completas
- Preservar nomes próprios e termos técnicos
- Manter tom natural mas profissional
- NÃO interpretar ou inferir — apenas limpar e estruturar

## Settings UI

Nova seção "Pré-processamento de Transcrições (Gemini)" em SettingsView:

- **Campo:** Google AI API Key (type=password, placeholder="AIza...")
- **Toggle:** "Ativar pré-processamento Gemini"
  - disabled quando API key vazia
  - hint dinâmico: "Reduz consumo de tokens do Claude em ~60%"
- **Label informativo:**
  - "O Gemini Flash limpa transcrições antes de enviar ao Claude, removendo ruído e repetições."
  - Link para aistudio.google.com/app/apikey

## Critérios de Sucesso

- [ ] Pré-processamento ativável via SettingsView
- [ ] Redução média de 50%+ no tamanho dos transcripts
- [ ] Qualidade preservada (passes subsequentes funcionam igual)
- [ ] Sem regressões no fluxo atual (toggle off = comportamento original)
- [ ] Logs mostrando economia de tokens por ingestão

## Riscos e Mitigações

| Risco | Mitigação |
|-------|-----------|
| Perda de informação no pré-processamento | Prompt conservador — só remove ruído, não interpreta |
| Latência extra do Google AI | Timeout curto (10s), fallback para texto bruto se falhar |
| API key inválida | Validação no save, fallback silencioso |
| Custo do Gemini | Flash é muito barato (~$0.10 por 1M tokens vs Claude) |
