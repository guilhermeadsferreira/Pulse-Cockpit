# Phase 999.5: Visibilidade do Board de Sustentação — Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Módulo standalone de Sustentação no app — tela própria (como Dashboard/Relatórios), não contida dentro do perfil de uma pessoa. O módulo observa o board Jira de suporte (atendimento/chamados de clientes) de forma independente: exibe métricas do board inteiro, executa análise de IA para extrair oportunidades e padrões. Os **insumos gerados** (insights, oportunidades, alertas) fluem para o perfil das pessoas por assignee — enriquecendo o metricas.md e/ou ações sugeridas do dev que trabalha com sustentação.

Escopo desta phase: configuração do board de sustentação, busca e exibição de métricas, análise de IA manual, persistência de insumos no perfil da pessoa.

</domain>

<decisions>
## Implementation Decisions

### Arquitetura do módulo
- **D-01:** SustentacaoView é uma tela top-level no app — entrada própria na navegação (sidebar), assim como Dashboard ou Relatórios. NÃO é uma aba dentro do PersonView.
- **D-02:** O módulo exibe o board de sustentação com visão do time inteiro (todos os tickets), com agrupamento por assignee/tipo. Foco não é num único indivíduo.
- **D-03:** Os insumos (oportunidades, padrões detectados) fluem para o perfil da pessoa por assignee — quem aparece mais no board recebe os insights no seu metricas.md via bloco gerenciado (padrão MetricsWriter da Phase 999.7).

### Configuração do board
- **D-04:** Board de sustentação é um Jira Software separado (não JSM) — configurado via `jiraSupportBoardId` + `jiraSupportProjectKey` nas **Settings globais** do app (não por pessoa).
- **D-05:** AppSettings recebe dois novos campos: `jiraSupportBoardId?: number` e `jiraSupportProjectKey?: string`.
- **D-06:** SLA thresholds configuráveis por tipo de issue nas Settings — ex: Bug > 3 dias, Task > 7 dias. Formato: `jiraSlaThresholds?: Record<string, number>` (tipo → dias). Padrão se não configurado: todos os tipos com threshold de 5 dias.

### Métricas capturadas
- **D-07:** Métricas do board de sustentação:
  - **Volume:** total de tickets abertos, fechados no período (últimos 30 dias)
  - **Tipos frequentes:** distribuição por tipo de issue (top 5) e por label (top 5)
  - **SLA breach:** tickets abertos além do threshold configurado para seu tipo — contagem total + lista de tickets em breach (key, summary, idade em dias, assignee)
  - **Comentários recentes:** últimos 3 comentários de cada ticket em breach (base para IA detectar padrões)
- **D-08:** Dados buscados por toda a equipe (sem filtro de assignee) — visão do board inteiro.

### Refresh de dados
- **D-09:** Dados do board de sustentação são buscados automaticamente via ExternalDataPass (junto com os dados do board principal) quando a pessoa tem o módulo habilitado. Sem custo extra de schedule separado.
- **D-10:** Módulo expõe um botão "Atualizar" para refresh manual também.

### Análise de IA
- **D-11:** Análise de IA é **manual** — botão "Analisar" na SustentacaoView. Não roda automaticamente a cada refresh de dados.
- **D-12:** O prompt recebe: lista de tickets (summary, tipo, labels, idade, comentários recentes, assignee). Extrai:
  - Padrões recorrentes (tipos de problema mais frequentes)
  - Oportunidades de automação (o que poderia ser resolvido sem intervenção humana)
  - Causa raiz mais provável dos tickets mais frequentes
  - Sugestões de redução de SLA (o que atrasa a resolução)
- **D-13:** Resultado da análise é persistido no metricas.md da pessoa com **mais tickets** (top assignee) — bloco gerenciado `SUSTENTACAO:ANALISE` via padrão MetricsWriter. Se há múltiplos assignees com tickets significativos, cada um recebe os insights relevantes para seus tickets.

### Claude's Discretion
- Layout interno da SustentacaoView (cards vs lista vs tabela)
- Localização exata na sidebar (ícone, ordem)
- Modelo de dados interno do `SupportSnapshot` (nome de campos)
- Threshold exato para "assignee com tickets significativos" (ex: mínimo 3 tickets)
- Se insumos vão também pro ActionRegistry (além do metricas.md) — decidir conforme complexidade

</decisions>

<specifics>
## Specific Ideas

- "Sustentação é uma coisa, daily de produto é outra — quero visões diferentes mas olhando cross" — o módulo deve ser claramente separado da visão de produto, mas os insights devem cruzar (ex: Edu tem X tickets de sustentação + Y issues no sprint → capacidade impactada).
- Oportunidades de automação e resolução de causa raiz são o valor principal da IA aqui — não só listar tickets.
- "Quero observabilidade para me antecipar" — alertas quando SLA está estourando devem ser visíveis antes de virar crise.
- Comentários do time de atendimento E do dev são relevantes para a análise — capturar ambos.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `JiraClient.searchIssuesByAssignee(email, jql?)`: reutilizável para buscar tickets sem filtro de assignee (passando JQL customizado por projectKey)
- `JiraClient.getIssueComments(issueKey, maxComments)`: já implementado — usar para capturar comentários dos tickets em breach
- `MetricsWriter` (Phase 999.7): padrão de blocos gerenciados com `<!-- METRICAS:SECTION_NAME -->` — exatamente o padrão para persistir insights de sustentação no metricas.md
- `ExternalDataPass.ts`: ponto natural de integração para fetch automático do board de sustentação

### Established Patterns
- Rate limiter no JiraClient (token bucket) — já protege contra 429
- Retry com backoff exponencial no JiraClient — já implementado
- Settings opcionais com `?:` — padrão do projeto para features configuráveis
- IPC handlers tipados com Zod/tipos explícitos — seguir padrão do index.ts
- Graceful degradation: falha na busca não impede resto do app (padrão de Phase 999.7)

### Integration Points
- ExternalDataPass.ts — adicionar fetch do support board junto com Jira/GitHub principal
- index.ts (main process) — novo IPC handler `get-sustentacao-data` e `run-sustentacao-analysis`
- Navegação do app (renderer) — nova entrada "Sustentação" na sidebar/nav

</code_context>
