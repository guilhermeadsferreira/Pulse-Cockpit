# Phase 999.11: Sustentação — Trending histórico + SLA compliance rate - Research

**Researched:** 2026-04-02
**Domain:** Electron/React — persistência de histórico JSON, cálculo de compliance, mini charts SVG
**Confidence:** HIGH

## Summary

Esta fase acrescenta memória temporal ao módulo de sustentação: em vez de exibir apenas um snapshot "agora", o app passa a acumular um snapshot por dia em `history.json` e a calcular métricas de tendência (deltas e SLA compliance rate). A implementação é puramente incremental — nenhum arquivo existente é substituído, apenas estendido.

O ponto de integração central é `fetchAndCacheSustentacao()` em `src/main/index.ts` (linha 1115). Após gravar `board.json`, a função também faz append/deduplica no `history.json`. O cálculo de compliance acontece no Main Process junto com o resto da lógica de negócio, e é retornado ao renderer dentro do payload já existente ou em um campo adicional.

Para os mini charts, nenhuma biblioteca de gráfico está instalada no projeto. A decisão (Claude's Discretion, ver abaixo) é entre instalar recharts, usar visx, ou renderizar SVG inline. A análise abaixo recomenda SVG inline — zero dependências, padrão existente no projeto (todos os indicadores visuais são inline styles), e volume de dados é trivial (90 pontos no máximo).

**Primary recommendation:** SVG inline para charts + compliance calculado no Main Process + history.json com deduplica diária. Zero novas dependências.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**D-01:** Snapshots acumulados em `history.json` no mesmo diretório de cache (`~/.pulsecockpit/cache/sustentacao/history.json`). Formato: array de snapshots com timestamp.

**D-02:** Granularidade diária — um snapshot por dia. Se múltiplos fetches no mesmo dia, manter o mais recente (deduplica por data ISO sem hora).

**D-03:** O `board.json` existente continua como cache de curto prazo (TTL 1h). O `history.json` é o acumulador de longo prazo, gravado a cada fetch bem-sucedido.

**D-04:** SLA compliance rate = (tickets resolvidos dentro do threshold / total tickets resolvidos) × 100, por janela temporal.

**D-05:** Duas janelas de compliance exibidas: 7 dias e 30 dias. A de 7d dá pulso rápido, a de 30d dá tendência.

**D-06:** Para calcular compliance, usar tickets com `statusCategory === 'done'` e `resolved` dentro da janela. Comparar `ageDias(created)` vs threshold do tipo no momento da resolução (tempo total aberto, não tempo em status específico).

**D-07:** Delta indicators nos cards existentes (Abertos, Fechados 30d, Breach): seta cima/baixo + valor absoluto vs snapshot de 7 dias atrás.

**D-08:** Dois novos cards no topo: "SLA Compliance 7d" e "SLA Compliance 30d" com percentual e delta.

**D-09:** Mini chart de evolução abaixo dos cards: breach count e compliance % ao longo do tempo (últimos 30 dias). Gráfico simples de linha.

**D-10:** 90 dias de snapshots diários retidos no `history.json`. Snapshots mais antigos são removidos automaticamente a cada gravação.

**D-11:** Sem compressão de snapshots antigos — o volume é pequeno (1 snapshot/dia × 90 dias ≈ 90 objetos JSON).

### Pattern 1: Append com deduplica diária (history.json)

**O que é:** Leitura → append/substituição → retenção → escrita síncrona. Exatamente o padrão do MetricsWriter mas para um único arquivo JSON.

**Lógica de deduplica:**
```typescript
// Chave de deduplica = data ISO sem hora (YYYY-MM-DD)
const dateKey = new Date().toISOString().slice(0, 10)

const existing = history.filter(e => e.date !== dateKey)
const updated = [...existing, { date: dateKey, snapshot, fetchedAt: Date.now() }]

// Retenção: manter apenas os últimos 90 dias
const cutoff = Date.now() - (90 * 24 * 60 * 60 * 1000)
const retained = updated.filter(e => e.fetchedAt >= cutoff)

writeFileSync(historyFile, JSON.stringify(retained, null, 0), 'utf-8')
```

**Graceful degradation:** try/catch envolve todo o bloco — falha no history nunca impede retorno do board.json.

### Pattern 2: Cálculo de SLA compliance no Main Process

**O que é:** Função pura que recebe o array de `JiraIssue` (já disponível dentro de `fetchSupportBoardMetrics`) e calcula compliance por janela.

```typescript
function calcularCompliance(
  issues: JiraIssue[],
  slaThresholds: Record<string, number>,
  windowDias: number
): number {
  const cutoff = Date.now() - windowDias * 24 * 60 * 60 * 1000
  const resolved = issues.filter(
    i => i.statusCategory === 'done' &&
    i.resolved !== null &&
    new Date(i.resolved).getTime() >= cutoff
  )

  if (resolved.length === 0) return 100 // sem dados = sem violação

  const compliant = resolved.filter(i => {
    const threshold = slaThresholds[i.type] ?? DEFAULT_SLA_DIAS
    const ageDias = Math.floor(
      (new Date(i.resolved!).getTime() - new Date(i.created).getTime()) / (1000 * 60 * 60 * 24)
    )
    return ageDias <= threshold
  })

  return Math.round((compliant.length / resolved.length) * 100)
}
```

**Nota crítica:** `JiraIssue.resolved` já está disponível (campo `resolutiondate` mapeado pelo JiraClient). Porém `SupportBoardSnapshot` atual **não inclui** os dados de compliance — é necessário estender o tipo ou retornar um envelope separado.

### Pattern 3: Delta indicator

**O que é:** Comparar valor atual com snapshot de 7 dias atrás no history.json.

```typescript
function getDelta(current: number, history: SustentacaoHistoryEntry[]): number | null {
  const sevenDaysAgo = Date.now() - 7 * 24 * 60 * 60 * 1000
  // Pega o snapshot mais próximo de 7 dias atrás
  const ref = history
    .filter(e => e.fetchedAt <= sevenDaysAgo)
    .sort((a, b) => b.fetchedAt - a.fetchedAt)[0]

  if (!ref) return null
  return current - ref.snapshot.ticketsAbertos  // exemplo para abertos
}
```

**Formato de exibição (recomendação):** `↑ 3` / `↓ 2` / `=` usando unicode + número absoluto. Mais compacto que "+3 vs semana passada" para cards pequenos.

### Pattern 4: Mini chart SVG inline

**O que é:** Função que recebe array de pontos e retorna JSX `<svg>`.

```typescript
function MiniLineChart({
  points,
  width = 200,
  height = 40,
  color = 'var(--accent)',
}: {
  points: number[]
  width?: number
  height?: number
  color?: string
}) {
  if (points.length < 2) return null
  const min = Math.min(...points)
  const max = Math.max(...points)
  const range = max - min || 1

  const coords = points.map((v, i) => {
    const x = (i / (points.length - 1)) * width
    const y = height - ((v - min) / range) * height
    return `${x.toFixed(1)},${y.toFixed(1)}`
  })

  return (
    <svg width={width} height={height} style={{ overflow: 'visible' }}>
      <polyline
        points={coords.join(' ')}
        fill="none"
        stroke={color}
        strokeWidth={1.5}
        strokeLinejoin="round"
      />
    </svg>
  )
}
```

**Dois charts:** breach count (cor: `var(--red)`) e compliance % (cor: `var(--accent)`). Ambos usam os últimos 30 pontos do history.

### Anti-Patterns a Evitar

- **Calcular compliance no renderer:** O renderer não tem acesso ao array de tickets resolvidos (apenas ao `SupportBoardSnapshot` resumido). O cálculo deve acontecer no Main Process onde `JiraIssue[]` está disponível.
- **Gravar history.json em todos os hits de cache:** O history só deve ser gravado quando um fetch real aconteceu (não em cache hit). O ponto correto é após `writeFileSync(cacheFile, ...)` no `fetchAndCacheSustentacao`.
- **Incluir `ticketsEmBreach` completo em cada snapshot do history:** O array de breach com comentários é pesado. Snapshots no history devem conter apenas números agregados — não a lista completa de tickets.

## Common Pitfalls

### Pitfall 1: compliance calcula sobre `SupportBoardSnapshot` em vez de `JiraIssue[]`

**O que vai errado:** `SupportBoardSnapshot` tem `ticketsFechadosUltimos30d` como número, não como array de tickets. Não há como calcular compliance (precisamos do `created` e `resolved` de cada ticket fechado).

**Por que acontece:** O developer lê o tipo `SupportBoardSnapshot` e assume que os dados necessários estão lá.

**Como evitar:** Calcular compliance **dentro de `fetchSupportBoardMetrics()`** antes de agregar, usando o `issues[]` completo que já está em memória. O resultado é adicionado ao `SupportBoardSnapshot` retornado (novos campos `complianceRate7d`, `complianceRate30d`).

**Sinal de alerta:** Qualquer código de compliance no renderer ou no IPC handler em vez de dentro de `SupportBoardClient.ts`.

### Pitfall 2: history snapshot inclui `ticketsEmBreach` completo

**O que vai errado:** history.json fica com centenas de KB em poucos dias (22 tickets × 90 snapshots com comentários = volumoso).

**Por que acontece:** Copiar `SupportBoardSnapshot` inteiro para o history sem filtrar.

**Como evitar:** Definir tipo separado `SustentacaoHistoryEntry` que omite `ticketsEmBreach` (ou guarda apenas `ticketsEmBreach.length`) e `recentComments`.

```typescript
export interface SustentacaoHistoryEntry {
  date: string          // YYYY-MM-DD
  fetchedAt: number     // timestamp ms
  ticketsAbertos: number
  ticketsFechadosUltimos30d: number
  breachCount: number   // apenas o número, não o array
  complianceRate7d: number
  complianceRate30d: number
}
```

### Pitfall 3: compliance = 100% quando não há dados na janela

**O que vai errado:** Exibir "100% compliance" enganosamente quando não há tickets fechados nos últimos 7 dias.

**Por que acontece:** `0 violações / 0 total = NaN` ou divisão por zero resulta em 100 ou Infinity.

**Como evitar:** Checar `resolved.length === 0` antes de dividir. Retornar `null` (não 100) quando não há dados, e exibir "—" na UI em vez de percentual.

### Pitfall 4: Delta indicator quando history tem menos de 7 dias

**O que vai errado:** Delta mostra `null` ou erro nos primeiros dias de uso.

**Por que acontece:** Não há snapshot anterior na janela de 7 dias.

**Como evitar:** Delta retorna `null` quando não há snapshot de referência. UI exibe "—" ou omite a seta. Não crashar.

### Pitfall 5: history.json corrompido bloqueia toda a view

**O que vai errado:** Se `history.json` tiver JSON malformado, o catch no IPC retorna erro e a SustentacaoView não carrega.

**Por que acontece:** Escrita incompleta (crash durante writeFileSync).

**Como evitar:** Wrap de leitura do history em try/catch separado do wrap do board.json. Se history falhar, retornar array vazio (não propagar erro).

### IPC handler para history

Duas opções de arquitetura:

**Opção A (simples):** Incluir os últimos 30 snapshots do history dentro do retorno de `sustentacao:get-data`. O renderer recebe tudo em uma chamada.

```typescript
// Estender SupportBoardSnapshot:
history: SustentacaoHistoryEntry[]  // últimos 30 dias

// Em fetchAndCacheSustentacao():
const historyData = readHistory(historyFile).slice(-30)
return { ...snapshot, history: historyData }
```

**Opção B (separado):** Novo IPC `sustentacao:get-history` que retorna o array completo. Renderer faz duas chamadas paralelas.

**Recomendação: Opção A.** Menos round-trips, menos código, alinhado com o padrão atual onde o get-data já retorna tudo que a view precisa.

// APÓS a linha: writeFileSync(cacheFile, JSON.stringify({ data, fetchedAt: Date.now() }), 'utf-8')
// ADICIONAR:
try {
  const historyFile = join(cacheDir, 'history.json')
  const dateKey = new Date().toISOString().slice(0, 10)

  if (existsSync(historyFile)) {

  const filtered = history.filter(e => e.date !== dateKey)

    date: dateKey,
    fetchedAt: Date.now(),

## Open Questions

1. **compliance quando ticket fechado mesmo dia que aberto**
   - O que sabemos: `ageDias` calcula floor(ms / 86400000) — ticket fechado no mesmo dia = 0 dias
   - O que é incerto: 0 dias <= threshold 5 dias = compliant. Parece correto, mas confirmar com o usuário se "fechado no mesmo dia" deve contar ou ser excluído
   - Recomendação: tratar como compliant (0 <= threshold = true), documentar no comentário do código

2. **Exibir compliance quando history.json ainda não tem dados**
   - O que sabemos: compliance7d/30d são calculados a partir dos tickets do fetch atual (não do history), então estarão disponíveis desde o primeiro fetch
   - O que é incerto: os **deltas** de compliance ficarão `null` até acumular 7 dias de history
   - Recomendação: UI exibe "—" para delta quando `null`, percentual sem delta é suficiente para mostrar valor desde o dia 1

3. **global.d.ts não tem `sustentacao` declarado**
   - O que sabemos: `window.api.sustentacao` funciona em runtime (preload expõe), mas TypeScript no renderer não valida a chamada
   - O que é necessário: adicionar bloco `sustentacao` ao `Window.api` em `global.d.ts` (deveria ter sido feito na Phase 999.5)
   - Recomendação: incluir como task desta fase — corrige dívida técnica existente
