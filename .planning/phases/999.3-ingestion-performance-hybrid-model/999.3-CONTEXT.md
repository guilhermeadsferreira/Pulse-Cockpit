# Phase 999.3: Performance de Ingestão + Modelo Híbrido — Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Avaliar e reduzir a latência do pipeline de ingestão através de um experimento controlado por estágios: substituir o Claude Code CLI pelo OpenRouter (modelo leve via REST) nos passes mais simples, começando pelo Pass Cerimônia.

**Escopo fixo — Estágio 1 (esta fase):**
- Instrumentação de latência: medir `elapsed` por pass (logs já existem no ClaudeRunner)
- Nova rota REST no `ClaudeRunner` para chamadas via OpenRouter (paralela à rota CLI existente)
- Pass Cerimônia (`cerimonia-sinal`) migrado para OpenRouter quando híbrido ativo
- Settings: campo `openRouterApiKey` + toggle "Usar modelo híbrido" na SettingsView
- Comparação de qualidade: output do Pass Cerimônia via OpenRouter vs Claude CLI

**Fora do escopo desta fase:**
- Pass 1 com modelo leve — Estágio 2, fase futura se Estágio 1 confirmar viabilidade
- Pass 1:1 deep — prompt recém-refinado, estabilizar antes de mexer no modelo
- Pass 2 — qualidade crítica, não tocar
- Substituição total do Claude Code CLI
- Novo UI de relatório de performance (logs de terminal são suficientes)

</domain>

<decisions>
## Implementation Decisions

### Provider e estratégia de integração
- **D-01:** OpenRouter via REST direto — sem SDK obrigatório, chamada HTTP com `Authorization: Bearer {key}`. Não usar Anthropic SDK ou API direta na primeira iteração.
- **D-02:** Abordagem por estágios: Estágio 1 = Pass Cerimônia. Estágio 2 (futuro) = Pass 1. Não comprometer tudo de uma vez — validar qualidade a cada estágio antes de avançar.
- **D-03:** Modelo inicial sugerido: GLM-4-Flash (via OpenRouter) — ~25× mais barato que Haiku, latência baixa, output estruturado compatível com o que o Pass Cerimônia espera. Pode ser configurável.

### API key management
- **D-04:** API key armazenada em `settings.json` (mesmo padrão do `claudeBinPath`) — campo `openRouterApiKey?: string` no `AppSettings`.
- **D-05:** Campo na SettingsView para o usuário colar a key. Plaintext no disco é aceitável para o perfil de usuário atual (gestor técnico, uso pessoal).

### Ativação do modelo híbrido
- **D-06:** Toggle explícito na SettingsView — "Usar modelo híbrido (OpenRouter)". Só ativa quando `openRouterApiKey` presente E toggle ligado.
- **D-07:** Degradação graciosa: se key ausente ou toggle desligado → comportamento atual (Claude CLI) sem erro. Se chamada OpenRouter falhar → fallback para Claude CLI com log de warning.
- **D-08:** Os passes elegíveis para OpenRouter são determinados pelo campo `useHybridModel` no settings — não hardcoded no pipeline.

### Quais passes são elegíveis
- **D-09:** Pass Cerimônia (`cerimonia-sinal`) é o único passe elegível nesta fase. Motivo: maior multiplicador de latência (N spawns por artefato coletivo), output estruturado simples, sem contexto de perfil.
- **D-10:** ClaudeRunner recebe nova função `runOpenRouterPrompt(apiKey, model, prompt, timeoutMs)` — paralela a `runClaudePrompt`, não substituta. O pipeline escolhe qual rota usar baseado no settings.

### Claude's Discretion
- Modelo específico a usar no OpenRouter (GLM-4-Flash como sugerido, mas planner pode escolher baseado em pesquisa atualizada de preço/qualidade)
- Estrutura exata do payload REST para OpenRouter (seguir documentação oficial)
- Como expor o resultado da comparação ao usuário (pode ser só log, pode ser toast na UI)

</decisions>

## Existing Code Insights

### Reusable Assets
- `ClaudeRunner.runClaudePrompt(claudeBin, prompt, timeoutMs, maxRetries?, model?)` — padrão a seguir para a nova `runOpenRouterPrompt`
- `SettingsManager.AppSettings.ingestionModel?: string` — precedente de campo de modelo no settings; usar mesmo padrão para `openRouterApiKey` e `useHybridModel`
- `ClaudeRunner` já loga `elapsed` em ms por spawn — base para comparação de latência sem instrumentação extra

### Established Patterns
- Settings em `~/.pulsecockpit/settings.json` via `SettingsManager` — não criar novo arquivo de config
- IPC tipado para settings (SettingsView lê/escreve via IPC) — nova key e toggle seguem o mesmo contrato
- Fallback explícito com log: padrão do pipeline quando algo falha (ver tratamento de erro em IngestionPipeline)

### Integration Points
- `IngestionPipeline.ts` método que chama Pass Cerimônia em batch (~linha 291): é aqui que `runClaudePrompt` vira condicional — `settings.useHybridModel ? runOpenRouterPrompt(...) : runClaudePrompt(...)`
- `SettingsView.tsx` — já tem seção de configurações de ingestão (campo do modelo); adicionar key + toggle junto

</code_context>

<specifics>
## Specific Ideas

- "Abordagem por estágios" foi explícita: o usuário quer validar Pass Cerimônia antes de comprometer Pass 1. Não adiantar estágios mesmo que pareça óbvio.
- Fallback automático para Claude CLI quando OpenRouter falha é requisito de robustez, não nice-to-have.
- Toggle explícito foi preferido sobre feature flag implícita — o usuário quer controle consciente sobre quando o híbrido está ativo.

</specifics>
