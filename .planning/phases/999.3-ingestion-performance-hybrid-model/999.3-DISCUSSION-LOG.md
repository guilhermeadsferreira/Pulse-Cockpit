## Modelo leve: qual provider?

**Question:** Claude Code CLI com modelo leve vs OpenRouter via REST vs Anthropic API direta?

**Options presented:**
- A) Claude Code CLI com modelo mais leve (`--model claude-haiku-4-5`) — zero nova dependência
- B) OpenRouter via REST — mais barato (~25×), sem CLI overhead, nova API key
- C) Anthropic API direta (Haiku) — sem CLI overhead, qualidade conhecida, nova dependência

**User response:** Não quer ir direto para OpenRouter. Quer avaliar em estágios, identificar oportunidades antes de comprometer.

**Captured decision:** Abordagem por estágios. Estágio 1 = Pass Cerimônia via OpenRouter. Estágio 2 (futuro) = Pass 1. Passes críticos (Pass 2, 1:1 deep) ficam no Claude CLI.

---

## API key: onde e como gerenciar?

**Question:** Settings.json + SettingsView, variável de ambiente, ou Keychain do macOS?

**Options presented:**
- A) Campo na SettingsView (consistente com claudeBinPath)
- B) Variável de ambiente (zero fricção de implementação)
- C) Keychain do sistema (mais seguro, mais trabalho)

**User response:** A — Campo na SettingsView.

**Captured decision:** `openRouterApiKey?: string` no AppSettings/settings.json, campo na SettingsView para colar a key.

---

## Quais passes ganham modelo leve + como ativar?

**Question:** Feature flag implícita (key presente → ativa), toggle explícito na SettingsView, ou sempre quando key disponível?

**Options presented:**
- A) Feature flag no settings — presença da key ativa automaticamente
- B) Toggle explícito na SettingsView — switch "Usar modelo híbrido"
- C) Sempre OpenRouter quando key disponível — sem toggle

**User response:** B — Toggle explícito.

**Captured decision:** Toggle "Usar modelo híbrido" na SettingsView. Só ativa quando key presente E toggle ligado. Fallback automático para Claude CLI se OpenRouter falhar.
